## Primary Keys

A primary key is a column that uniquely identifies each row in a table.

Every important table has its own primary key:

- customers.customer_id
- products.product_id
- orders.order_id
- order_items.order_item_id

Primary keys prevent duplicate records and make it possible to connect tables correctly.


## Foreign Keys

A foreign key is a column that references a primary key from another table.

Foreign keys create relationships between tables and protect data quality.

Relationships in this database:

- orders.customer_id references customers.customer_id
- order_items.order_id references orders.order_id
- order_items.product_id references products.product_id

For example, an order cannot be created for a customer that does not exist.


## Auto Increment

Auto Increment automatically generates unique IDs when a new record is inserted.

Example:

```sql
INSERT INTO customers (customer_name, city, segment)
VALUES ('Arta Krasniqi', 'Vushtrri', 'Business');

The database automatically creates:

customer_id = 1
```
34. What happened when you inserted a customer without specifying customer_id?

The database automatically generated the customer_id because the column uses AUTOINCREMENT.

35. Why is AUTOINCREMENT useful?

AUTOINCREMENT is useful because it creates unique IDs automatically.

It prevents manually creating duplicate IDs and makes inserting new records easier and safer.

36. Should we manually choose IDs in a real database? Why or why not?

No, IDs should usually be generated by the database.

Manually choosing IDs can create duplicates, conflicts, and data quality problems.

37. What happens if we delete a row? Does AUTOINCREMENT reuse the old ID?

No. AUTOINCREMENT does not reuse deleted IDs.

Example:

If customer_id 5 is deleted, the next inserted customer will receive a new ID instead of using 5 again.

38. Why is a stable unique ID better than using customer_name as the identifier?

Names are not reliable identifiers because:

Multiple customers can have the same name.
Names can change.
Names are text values and slower for relationships.

A unique ID is stable and designed for database relationships.

## Relationships

The database contains these relationships:

One-to-Many Relationship

One customer can have many orders.

customers
     |
     |
   orders

Example:

One customer can create multiple orders.

Many-to-Many Relationship

Orders and products have a many-to-many relationship.

One order can contain many products.

One product can appear in many orders.

Because of this, the order_items table is used as a bridge table.

orders
   |

   |

## Valid and Invalid Insert Tests

Foreign keys and constraints were tested by intentionally inserting invalid data.

Examples of rejected inserts:

Invalid customer_id

Trying to create an order with a customer that does not exist:

INSERT INTO orders
(customer_id, order_date, status, channel)
VALUES
(99, '2026-07-10', 'completed', 'Online');

Result:

The database rejected the insert because customer_id 99 does not exist.

Invalid product price

Trying to insert a product with negative price:

INSERT INTO products
(product_name, category, price)
VALUES
('Charger', 'Accessories', -2);

Result:

Rejected because price must be greater than 0.

Invalid order quantity

Trying to insert quantity 0:

INSERT INTO order_items
(order_id, product_id, quantity)
VALUES
(1,1,0);

Result:

Rejected because quantity must be greater than 0.

## INNER JOIN vs LEFT JOIN
INNER JOIN

INNER JOIN returns only records that have matching values in both tables.

Example:

If we join customers and orders using INNER JOIN, only customers who have at least one order will appear.

Customers without orders are excluded because there is no matching record in the orders table.

Example:

SELECT
    customers.customer_name,
    orders.order_id
FROM customers
INNER JOIN orders
ON customers.customer_id = orders.customer_id;
LEFT JOIN

LEFT JOIN returns all records from the left table and matching records from the right table.

If there is no match, NULL values are returned.

Example:

Showing all customers including customers who never created an order.

LEFT JOIN is useful when we want to see all records from the main table even if related data does not exist.

Important business rule:

Completed revenue only includes orders where:

status = 'completed'

Pending and cancelled orders appear in operational reports but are not included in revenue calculations because they are not completed sales.
